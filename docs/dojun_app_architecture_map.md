# 1. 전체 시스템 한눈에 보기

```text
[관리자 웹]
   ├─ 공지 작성 / 수정
   ├─ 투표 생성 / 수정
   ├─ 설문 생성 / 수정
   ├─ 이미지 업로드
   ├─ 상태 변경(open / closed / archived)
   └─ 관리자 운영

            │
            ▼

[Firebase]
   ├─ Firestore
   │   ├─ issues_public
   │   ├─ issues_private
   │   ├─ votes
   │   ├─ vote_stats
   │   ├─ admins
   │   └─ meta/version
   │
   ├─ Storage
   │   └─ 공지 이미지 / 자료 파일
   │
   └─ 인증 / 보안 규칙
       ├─ admin role
       ├─ read / write rules
       └─ 앱 버전 통제

            │
            ▼

[모바일 앱]
   ├─ 홈
   ├─ 공지
   ├─ 투표
   ├─ 설문
   ├─ 자료실
   ├─ 강제 업데이트
   └─ 알림 수신

            │
            ▼

[조합원 700명]
   ├─ 공지 확인
   ├─ 교섭 브리핑 열람
   ├─ 투표 참여
   ├─ 설문 참여
   ├─ 자료 다운로드
   └─ 알림 수신
```

---

# 2. 관리자 웹 역할

관리자 웹은 **데이터 생성·수정의 중심**이다.

## 주요 기능
- 공지 작성 / 수정
- 공지 이미지 업로드
- 투표 생성 / 수정
- 설문 생성 / 수정
- 상태 관리
- 보관 / 복구
- 순서 정렬
- 관리자 권한 관리

## 운영 원칙
- `draft` : 작성 중
- `review` : 내부 검토
- `open` : 조합원 공개
- `closed` : 종료
- `archived` : 보관

즉, 관리자 웹은 **콘텐츠를 만들고 공개 상태를 통제하는 도구**다.

---

# 3. Firebase 역할

Firebase는 dojun_app의 **중앙 허브**다.

## Firestore 컬렉션 구조

### `issues_public`
공지 / 투표 / 설문 본문 데이터

주요 필드
- type
- title
- summary
- content
- imageUrl
- options
- multiple
- maxSelections
- status
- active
- createdAt
- updatedAt

### `issues_private`
내부 검토용 데이터  
모바일 앱에서는 읽지 않음

### `votes`
투표 / 설문 메타 정보

### `vote_stats`
집계 결과  
모바일 앱은 읽기 전용

### `admins`
관리자 계정 및 권한
- viewer
- editor
- super_admin

### `meta/version`
강제 업데이트 기준 문서
- latestVersion
- minimumVersion
- updateRequired
- message

---

# 4. Storage 역할

Firebase Storage는 **파일 저장소**다.

## 저장 대상
- 공지 이미지
- 자료실 PDF
- 향후 첨부파일

## 현재 역할
- 관리자 웹에서 이미지 업로드
- 다운로드 URL을 Firestore에 저장
- 모바일 앱에서 이미지 표시
- 이미지 제거 시 실제 Storage 파일 삭제

---

# 5. 모바일 앱 역할

모바일 앱은 **조합원이 실제로 사용하는 사용자 화면**이다.

## 핵심 기능
- 공지 열람
- 이미지 공지 표시
- 투표 참여
- 설문 참여
- 결과 확인
- 자료실 열람
- 앱 강제 업데이트
- 수동 / 자동 refresh

## 표시 규칙

### 공지
- `status in ["open", "closed"]`
- `active == true`

### 투표 / 설문
- `status == "open"`
- `active == true`

즉 모바일 앱은 **관리자 웹에서 공개된 데이터만 보여준다.**

---

# 6. 데이터 흐름

## 공지 흐름

```text
관리자 웹에서 공지 작성
→ Firestore issues_public 저장
→ 이미지가 있으면 Storage 업로드
→ 모바일 앱이 issues_public 읽음
→ 조합원이 공지 확인
```

## 투표 흐름

```text
관리자 웹에서 투표 생성
→ issues_public 저장
→ votes 메타 문서 생성
→ 모바일 앱에서 공지 + 투표 정보 표시
→ 조합원이 참여
→ ballots 저장
→ vote_stats 집계
→ 모바일 앱 / 관리자 웹 결과 표시
```

## 설문 흐름

```text
관리자 웹에서 설문 생성
→ issues_public 저장
→ votes 메타 문서 생성
→ 모바일 앱에서 설문 표시
→ 조합원 응답
→ 결과 집계
→ 관리자 / 모바일 앱 반영
```

---

# 7. 보안 구조

## 관리자 권한
- `viewer` : 읽기 중심
- `editor` : 생성 / 수정 가능
- `super_admin` : 완전 삭제 및 관리자 관리 가능

## 조합원 권한
- 공개 데이터 읽기
- 본인 투표 참여
- 결과 읽기

## Rules 목적
- 관리자 외 문서 수정 차단
- vote_stats 클라이언트 write 차단
- 앱 버전 기준 통제
- 공개 데이터와 비공개 데이터 분리

# 8. 밴드 대체 전략에서의 위치

dojun_app은 단순 앱이 아니라 **노조 운영 인프라**다.

기존 BAND 역할
- 공지
- 간단한 소통
- 자료 전달

dojun_app의 확장된 역할
- 공지
- 교섭 브리핑
- 투표
- 설문
- 자료실
- 푸시 알림
- 버전 통제
- 운영 데이터 축적

즉 dojun_app은 BAND보다 **더 정리되고, 더 통제되고, 더 참여적인 플랫폼**이다.
